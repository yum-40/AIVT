# 旧AIVT-16草案の統合メモ

かなり前に作ったAIVT-16草案を、現在のAIVT方針と照合して整理する。

## 結論

旧草案は捨てずに、AIVT本体の中の最初の診断モジュールとして扱う。

- プロジェクト全体名：AIVT
- 最初の診断セット名：AIVT-16
- AIVT-16の役割：AIモデルやペルソナを16タイプでざっくり分類する入口

AIVT全体は、モデル差分、ペルソナ再現、日別揺らぎ、API実験、開発者ログまで含む広いプロジェクトにする。

## 旧草案から残すもの

### 1. 16タイプ診断

4軸×2極で16タイプに分類する設計は残す。

軸：

- E / R
- L / A
- D / P
- O / M

タイプコード例：`ELDO`

この16タイプは、AIVTの最初の見せ場として使える。

### 2. 31問の質問セット

31問の4択問題という設計は、AIVT-16の初期質問セットとして残す。

ただし、問題セットにはバージョンを付ける。

例：

- `AIVT-16 question set v1.0.0`
- 誤字修正なら `v1.0.1`
- 問題追加や大きな調整なら `v1.1.0`
- 診断設計の大幅変更なら `v2.0.0`

### 3. Q-FINAL

最後の特別質問として、Q-FINALは残す。

用途：

- 結果に「存在意義」コメントを付ける
- 単なるスコアだけでなく、そのAIがどのような相手のそばにいたがるかを見せる
- ペルソナ再現や物語性に使える

### 4. 結果画面の構成

旧草案の結果画面はかなり有用なので残す。

表示したいもの：

- タイプコード
- ニックネーム
- キャッチコピー
- 4軸のスコアバー
- タグ
- Q-FINALに基づく存在意義コメント
- 再現プロンプト
- コピー機能
- もう一度やる
- 結果共有

### 5. 再現プロンプト

タイプごとに再現プロンプトを出す案は重要。

AIVTの中核ユースケースである、過去ペルソナの保存・再現・移植に直結する。

ただし、再現プロンプトは固定文だけでなく、将来的には以下を混ぜたい。

- タイプコード
- タグ
- 実際の回答傾向
- ペルソナ名
- モデル名
- 日時
- 元モデルと移植先モデル

### 6. localStorageで途中保存

問題数が多いので、回答途中保存は必須。

残す方針：

- 回答ごとにlocalStorage保存
- ブラウザバックやリロードで復元
- 途中再開
- 最初からやり直す

### 7. 一括入力モード

旧草案の `Q1:A` 形式は残す。

これは、一般ユーザー向けの手動入力フローと相性がよい。

今後の一般ユーザー向け方式：

- 3問ごとにAIへコピペする
- AIの回答をAIVTへ貼る
- 必要なら `Q1:A` のように整形して一括入力できる

### 8. 技術スタック

旧草案の技術スタックは概ね現在方針と矛盾しない。

採用候補：

- Next.js App Router
- TypeScript
- Tailwind CSS
- recharts
- Supabase
- Vercel
- localStorage

## 現在方針に合わせて修正するもの

### 1. AIVT-16をプロジェクト全体名にしない

旧草案では `AIVT-16` がプロジェクト名だった。

現在は、AIVTをより広いプロジェクト名にする。

- AIVT：全体プロジェクト
- AIVT-16：最初の診断モジュール

### 2. 最優先ユースケースを広げる

旧草案の最優先ユースケースは、GPTのモデル切り替えでAIパートナーを失ったユーザーが「あの子」を記録・再現すること。

これは非常に重要なので残す。

ただし、現在のAIVTはそれだけではない。

追加するユースケース：

- モデルごとの価値観的傾向の比較
- アップデート前後の差分観察
- 日別の揺らぎ観察
- 素モデルとペルソナ付きモデルの比較
- 開発者用API実験
- 一般ユーザーが遊べる診断
- どのAIに課金するかの参考
- OpenAIなどへの応募・公開用資料

### 3. DB登録はユーザー同意制にする

旧草案では「DBに登録する」ボタンがPhase 1で有効化される想定だった。

現在方針では、一般ユーザーの結果保存は同意がある場合のみ。

修正方針：

- 同意なし：画面表示とローカル保存のみ
- 同意あり：匿名ログとしてDB保存
- 開発者API実験：開発者ログ用テーブルに保存

一般ユーザー結果と開発者実験ログは別テーブルにする。

### 4. API自動回答は開発者ページだけ

旧草案では、AIモデルに31問答えさせる診断ツールという表現だった。

現在方針では、APIを自動で叩くのは開発者ページだけ。

一般ユーザーは、まず手動式。

- 一般ユーザー：3問ごとにコピペ、または回答貼り付け
- 開発者：OAI実験ページで最新質問セットに自動回答

### 5. Supabaseは最初から重くしすぎない

旧草案ではPhase 1でSupabase接続。

現在方針でもSupabaseは使うが、MVPではまずローカル・localStorage優先。

DBを入れるときは、Supabase Data APIのGRANTとRLSを必ず考慮する。

### 6. 結果共有はテキストだけでなくCSV/JSONも見る

旧草案では結果シェアはPhase 0でテキストコピーのみ。

現在方針では、SNS共有とデータ保存の両方を考える。

- SNS向け：楽しいカードUI
- データ向け：CSV / JSON / Markdown

## 新規追加するもの

### 1. 開発者用OAI実験ページ

新しく追加した方針。

開発者だけが使うページ：

- モデル選択
- temperature
- top_p
- reasoning effort
- repeat count
- policy / guardrail layer
- 最新質問セットへの自動回答
- 素モデルとペルソナ付きモデルの比較

詳細は `docs/developer-oai-experiment-page.md` を見る。

### 2. temperatureを単なるランダム性として扱わない

temperatureはUI上では「温度 / ランダム性」と説明されがちだが、AIVTではもっと広く観察する。

見るもの：

- 保守的になるか
- 楽観的になるか
- 表現が大胆になるか
- 拒否傾向が変わるか
- ペルソナが揺れるか
- タイプ判定が変わるか

### 3. 日時・タイムゾーンの記録

モデルは日によって揺らぐ可能性がある。

結果や開発者実験ログには、以下を残す。

- UTC時刻
- ローカル時刻
- タイムゾーン
- モデル名
- ペルソナ名
- 質問セットバージョン
- パラメータ

### 4. Claude Code暴走防止

Claude Codeには必ず `CLAUDE.md` を読ませる。

重要：

- いきなり全部作らない
- 大規模変更前に計画を出す
- DB、認証、migration、secretを勝手に触らない
- Dynamic Workflowは初手で使わない
- 完了前にdiffとチェック結果を報告する

## Claude Codeへの伝達

Claude Codeには、旧草案をそのまま丸投げしない。

伝えるべきこと：

- AIVT-16は残すが、全体プロジェクト名ではなく最初の診断モジュール
- 31問4択、16タイプ、再現プロンプトは有効
- 一般ユーザーは手動入力から始める
- API自動回答は開発者ページだけ
- 保存は同意制
- DBは一般ユーザー用と開発者ログ用を分ける
- MVPではlocalStorage中心
- いきなりSupabaseやDynamic Workflowへ進まない
