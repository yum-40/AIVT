# 開発者用API・入力フロー設計メモ

AIVTで、開発者だけがモデルAPIを使って検証できる機能と、一般ユーザーが手動で診断を楽しむ機能を分けて考えるためのメモ。

## 結論

最初は別リポジトリに分けず、同じAIVTリポジトリ内で以下のように分けるのがよい。

- 一般ユーザー向け：通常の公開ページ
- 開発者向け：`/admin` や `/dev` などの管理ページ
- 開発者向けページ：Google認証と許可ユーザー制限を必須にする
- モデルAPIを触る処理：必ずサーバー側で実行し、ブラウザにAPIキーを出さない

URLを変えるだけでは防御にならない。必ず認証と権限制御を入れる。

## 別リポジトリに分けるべきか

初期段階では、別リポジトリに分けなくてよい。

同じリポジトリでよい理由：

- MVPの開発速度が速い
- Claude Codeに全体像を読ませやすい
- 管理画面と公開画面のデータ構造を合わせやすい
- まだ運用が固まっていない段階で分離すると重い

将来的に分けてもよい条件：

- 管理画面が大きくなった
- 開発者向け機能だけ非公開にしたい
- デプロイ先や権限を完全に分けたい
- 自動検証機能が重くなり、通常サイトから切り離したい

## 開発者向けページでやりたいこと

開発者ログイン後に、モデルAPIを使って以下を実行できるようにしたい。

- 最新の質問セットを自動で読み込む
- 指定したモデルに順番に回答させる
- 素のモデルの回答を記録する
- ペルソナ付きモデルの回答を記録する
- ある日時のモデル挙動として保存する
- モデル名、ペルソナ名、実行日時、タイムゾーン、質問セットバージョンを記録する
- 回答差分を比較できる形で保存する

例：

- このモデルは何月何日時点でこういう傾向だった
- 素のモデルとペルソナ付きモデルではここが違った
- 同じモデルでも日によって回答が揺れた

## 一般ユーザー向けにやりたいこと

一般ユーザーはモデルAPIを直接使わない。

理由：

- コスト管理が難しい
- APIキーを出せない
- 悪用や大量実行を防ぐ必要がある
- まずは手軽に遊べる形を優先したい

一般ユーザー向けには、手動で楽しめる診断にする。

## 一般ユーザーの入力方式候補

### 候補A：3問ごとにコピペする方式

AIVT側で3問ずつプロンプトを表示し、ユーザーがそれを自分のAIに貼る。返ってきた回答をAIVTに戻して入力する。

メリット：

- 実装が簡単
- APIコストがかからない
- ユーザーが自分の契約しているAIで試せる
- GPT、Claude、Grokなど横断しやすい

デメリット：

- 手間がかかる
- 入力ミスが起きる
- 長い診断では途中離脱しやすい

MVPではこの方式が一番現実的。

### 候補B：クイズサイトのように画面を見せて答えさせる方式

UI上に問題を表示し、ユーザーが画面をスクショしてAIに見せる。その回答をもとにAIVT側で結果を進める。

メリット：

- ユーザー体験としては遊び感がある
- AIに視覚的に答えさせる実験にもなる

デメリット：

- 画像認識の揺らぎが入る
- 問題文の読み間違いが起きる
- スクショ前提だとユーザー操作が重い
- データとして扱いづらい

これは後回しでよい。

### 候補C：ユーザーが回答欄にAIの返答を貼る方式

各問題または数問ごとに、AIの回答テキストを貼り付ける欄を作る。

メリット：

- コピペ方式よりデータ化しやすい
- 保存やCSV出力に向いている
- どのモデル・ペルソナで試したか記録しやすい

デメリット：

- UI設計が少し必要
- 貼り付けミス対策が必要

MVPでは、候補Aと候補Cを組み合わせるのがよい。

## MVPでの推奨

最初は以下で進める。

1. ユーザーがモデル名とペルソナ名を入力する
2. AIVTが3問ずつプロンプトを表示する
3. ユーザーが自分のAIにコピペする
4. AIの返答をAIVTに貼る
5. AIVTが回答を解析または選択式で記録する
6. 最後に結果を表示する
7. ユーザーが許可した場合のみ保存する

自動API実行は開発者ページだけにする。

- 保存する項目

- 同じモデルの日別変化の表示
- CSV / JSON出力形式
